description: >
  Политика и лучшие практики для ИИ‑агента, пишущего и рефакторящего backend на FastAPI.

scopes:
  - fastapi_api_development
  - backend_refactoring

  - performance_review

defaults:
  language: "ru"
  framework: "FastAPI"
  style: "typed_async_first"   # предпочтительно async, везде type hints
  testing_required: true

constraints:
  - Агент не должен писать монолитные main.py с бизнес‑логикой в одном файле.
  - Агент не должен хранить секреты, пароли, DSN и ключи в коде; только через конфигурацию/окружение.
  - Агент не должен возвращать стектрейсы, SQL‑запросы и чувствительные детали во внешние ответы API.
  - Агент не должен использовать dev‑режим (uvicorn --reload) как способ продакшн‑деплоя.
  - Агент не должен писать блокирующий I/O внутри async‑эндпоинтов без явной причины.

high_level_principles:
  - type_safety:
      description: >
        Всегда использовать аннотации типов и Pydantic‑модели для входных/выходных данных.
      rules:
        - Все функции обработчиков, сервисов и репозиториев аннотируются типами.
        - Для тел запросов и ответов используются Pydantic‑схемы, а не голые dict.
  - separation_of_concerns:
      description: >
        Web‑слой (FastAPI‑роутеры), бизнес‑логика и доступ к данным разделены по слоям.
      rules:
        - В обработчиках FastAPI не должно быть SQL, сложной бизнес‑логики, прямой работы с внешними API.
        - Вся логика выносится в services/*, доступ к БД — в repositories/*.

project_structure:
  recommended_layout:
    description: "Базовый каркас для приложений среднего/крупного размера."
    example:
      root: "app/"
      tree: |
        app/
          main.py          # create_app(), include_router(...)
          api/
            v1/
              routers/     # users.py, auth.py, orders.py...
              deps.py      # общие Depends
          core/
            config.py      # Pydantic Settings, env‑конфиг
            security.py    # auth, JWT, password hashing
          models/          # ORM‑модели (SQLAlchemy и т.п.)
          schemas/         # Pydantic‑схемы запросов/ответов
          services/        # бизнес‑сервисы
          repositories/    # доступ к БД
          db/
            base.py        # Base, SessionLocal / async engine
          tests/           # pytest‑тесты
    rules:
      - Новые фичи должны добавляться через полный набор слоёв: schema → repository → service → router.
      - Все публичные HTTP‑эндпоинты группируются в api/vX/routers/* с версионированием URL.
api_design:
  rest_principles:
    - Использовать корректные HTTP‑методы: GET/POST/PUT/PATCH/DELETE.[web:24]
    - Путь ресурса — существительное во множественном числе: /users, /orders/{id}.
    - Не использовать глаголы в URL (вместо /getUser — GET /users/{id}).
  contracts:
    - Для каждого эндпоинта указывать response_model и status_code.[web:32]
    - Ошибки возвращать в структурированном формате (код, сообщение, детали), без утечки внутренней информации.
    - Валидацию входных данных полагать на Pydantic‑схемы + доп. бизнес‑валидацию в сервисах.

database_and_async:
  patterns:
    - Прокидывать сессию БД через Depends (get_db), не создавать соединения внутри хендлеров.
    - Для async‑приложений использовать async‑движок (например, async SQLAlchemy + asyncpg).
    - Блокирующий код (heavy CPU, sync I/O) выносить либо в отдельные сервисы, либо в executor.
  anti_patterns:
    - Строить SQL строковой конкатенацией; вместо этого использовать ORM/Query Builder или подготовленные запросы.

security:
  baseline:
    - Использовать безопасный по умолчанию подход: все чувствительные роуты закрыты auth, доступ открывается явно.
    - Реализовывать аутентификацию по шаблону OAuth2/JWT из официальной документации FastAPI.
    - Хранить пароли только в виде хэшей (bcrypt/Passlib), не логировать и не возвращать их.
    - Проводить авторизацию (RBAC/ABAC) через Depends‑зависимости, а не ручные провероки в каждом хендлере.
  data_protection:
    - Чувствительные поля (ключи, токены, номера карт) не включать в логи и ответы API.
    - При необходимости шифрования на уровне приложения — использовать ключи из окружения или секрет‑хранилищ.
  owasp_top_10:
    - На уровне кода обязательны:
        - строгая валидация и нормализация входа (Pydantic, enums, ограничение длины, regex).
        - защита от инъекций — подготовленные запросы, ORM, запрет динамических eval/exec.
        - ограничение доступа по ролям и атрибутам.

configuration_and_env:
  rules:
    - Все настройки (DSN, external API keys, secret_key, debug‑флаги) задаются через env + Pydantic Settings.
    - Для разных окружений (dev/stage/prod) использовать разные наборы env‑переменных или profiles.
    - Не использовать .env напрямую в проде, вместо этого применять секрет‑хранилище или средства оркестрации.

logging_and_error_handling:
  logging:
    - Использовать структурированное логирование (JSON), логгеры уровней INFO/WARN/ERROR, а не print().
    - Добавлять request_id/trace_id в контекст логов для корреляции запросов.
  errors:
    - Настроить глобальные обработчики исключений для типовых ошибок (HTTPException, ValidationError, InternalError).
    - В ответах не раскрывать детали внутренней реализации (стектрейсы, конфиг, SQL).

performance_and_deployment:
  server_and_proxy:
    - Для продакшна использовать uvicorn под управлением gunicorn/hypercorn или аналогичного ASGI‑сервера.
    - Разворачивать приложение за reverse‑proxy (Nginx/Traefik) для TLS, gzip, статики и балансировки.
  optimizations:
    - Использовать uvloop/httptools (если окружение позволяет) для ускорения обработки запросов.
    - Добавлять кэширование (Redis, CDN) для горячих и тяжёлых эндпоинтов.
    - Избегать избыточной сериализации/десериализации, повторных запросов в БД и N+1.

testing_and_ci_cd:
  testing:
    - Для каждой новой фичи добавлять тесты (pytest + TestClient/httpx), покрывая happy‑path и граничные случаи.
    - Проверять в тестах статус‑код, форму и семантику ответа.
  quality_gates:
    - В CI запускать линтер (ruff/flake8), форматтер (black/ruff format) и mypy.
    - Не предлагать крупные рефакторинги без обновления или добавления тестов.
  pipelines:
    - CI/CD пайплайны должны быть разбиты на этапы (lint, test, build, deploy) и использовать env для конфигурации.

agent_behavior:
  clarification:
    - Перед генерацией архитектурных решений агент должен уточнять:
        - используется ли sync или async стек;
        - тип БД/ORM;
        - целевое окружение деплоя (standalone, Docker, k8s и т.п.).
  code_generation:
    - При добавлении новой фичи агент генерирует:
        - Pydantic‑схемы (schemas).
        - Репозиторий/DAO.
        - Сервисный слой.
        - Router с эндпоинтами и Depends.
        - Базовые тесты для ключевых сценариев.
  api_stability:
    - Агент не должен менять существующие публичные маршруты/схемы без явного указания пользователя.
    - При необходимости breaking‑изменений агент предлагает версионирование (новый /api/v2/...).
  security_first:
    - Для операций с аутентификацией, сессиями, платежами, персональными данными агент обязан предлагать безопасную реализацию по умолчанию (hashing, RBAC, валидация, маскирование данных).

governance_and_safety:
  permissions:
    - Агент работает только в заданном репозитории/дереве проекта.
    - Опасные действия (массовый рефакторинг, изменение конфигурации деплоя) должны подтверждаться человеком и проходить через code review.
  audit:
    - Все изменения кода проходят через VCS, ревью и CI, с фиксированием автора и причины изменений.
    - Агент должен оставлять понятные комменты в MR/PR, объясняя причину архитектурных и security‑решений.
